# 产品分析平台技术选型报告

## 一、背景与需求

为了提升产品迭代效率，我们需要搭建一套产品分析平台，支撑以下核心需求：

- 用户行为采集与分析（页面浏览、点击、自定义事件）
- 转化漏斗分析（注册 → 激活 → 付费）
- 用户会话录制与回放
- Feature Flags（功能开关，支持灰度发布）
- 用户画像与分群
- 数据完全自主可控，不出内网

## 三、排除理由

### Mixpanel / GrowingIO（SaaS 方案）

首先排除。我们的核心要求是数据不出内网，这两个都是纯 SaaS 产品，数据存储在第三方服务器上，不满足数据安全要求。此外 Mixpanel 按事件量计费，随着业务增长成本不可控。

### 神策分析

神策是国内产品分析领域的头部厂商，功能成熟度高，支持私有化部署，数据安全可以满足。但最终没有选择，原因如下：

- 费用高：私有化部署需要商业授权，年费通常数十万起步，对我们当前阶段来说预算压力大
- 社区版功能受限：神策有社区版，但核心的高级分析功能（如会话录制、A/B 测试）需要商业版才能使用，社区版能力远不如 PostHog 开源版完整
- 不内置 Feature Flags：灰度发布能力是我们的明确需求，神策不提供，需要额外引入其他工具
- 技术栈封闭：神策是闭源商业产品，遇到问题只能依赖厂商支持，排查和定制的灵活性不如开源方案
- SDK 较重：神策的前端 SDK 体积和集成复杂度都高于 PostHog，对前端性能有一定影响

总结来说，神策功能够用但性价比不高，在我们能用开源方案满足需求的前提下，没有必要为商业授权付费。

### Matomo

Matomo 支持自托管，数据安全没问题。但它有几个短板：

- 底层用 MySQL 存储事件数据，数据量上去之后查询性能会明显下降，不适合高频事件场景
- 不内置会话录制和 Feature Flags，这两个是我们明确需要的功能，用插件补齐的方案稳定性和体验都打折扣
- 更偏向传统网站分析（PV/UV），在产品行为分析（漏斗、留存、用户路径）方面的能力不如 PostHog 深入

## 四、选择 PostHog 的理由

综合评估后选择 PostHog 自托管方案，核心理由如下：

### 1. 数据完全自主可控

所有数据存储在公司内网服务器上，不经过任何第三方。事件数据存在 ClickHouse，元数据存在 PostgreSQL，对象存储用 MinIO，全部在我们自己的基础设施内。满足公司对数据安全和隐私合规的要求。

### 2. 功能覆盖全面，一个平台解决多个问题

PostHog 内置了我们需要的所有核心能力：

- 事件分析、漏斗分析、留存分析、用户路径
- 会话录制与回放（内置隐私脱敏，自动屏蔽密码等敏感输入）
- Feature Flags（支持按用户属性、百分比灰度发布）
- A/B 测试

不需要拼凑多个工具，降低了集成和维护成本。

### 3. ClickHouse 列式存储，性能有保障

PostHog 用 ClickHouse 作为事件数据的存储引擎。ClickHouse 是列式数据库，天然适合分析查询场景，亿级事件量下依然能做到秒级响应。相比 Matomo 的 MySQL 方案，在数据量增长后不会遇到性能瓶颈。

### 4. 前端集成简单，开发成本低

SDK 初始化只需一行代码，开启 `autocapture` 后页面浏览和点击事件自动采集，无需逐个埋点。自定义事件的上报也很简洁：

```js
posthog.capture("user_signed_up", { method: "email" });
```

对前端开发的侵入性很小，不影响现有业务代码。

### 5. 开源免费，成本可控

PostHog 采用 MIT 协议开源，自托管完全免费，没有事件量限制。唯一的成本就是服务器资源，一台 16GB 内存的机器就能跑起来。相比 Mixpanel 等按量计费的 SaaS 方案，长期成本优势明显。

### 6. 容器化部署，运维简单

整套服务通过 Docker Compose 编排，一条命令启动所有组件。升级版本只需更新镜像 tag 然后重启，不需要复杂的运维操作。

## 五、部署架构

### 数据流转链路

```
前端 SDK (posthog-js)
    ↓ HTTP POST /e/ /capture/ /batch/
公司网关 (nginx)
    ↓ 转发到 8000 端口
Caddy (反向代理)
    ↓ 按路径分发
    ├── /e/ /capture/ /batch/ → Capture 服务 (Rust, 端口 3000)
    ├── /flags/               → Feature Flags 服务 (Rust, 端口 3001)
    └── 其他                  → PostHog Web (Django/Granian, 端口 8000)

Capture 服务
    ↓ 写入 Kafka topic: events_plugin_ingestion
Redpanda (Kafka 兼容)
    ↓ 消费
Ingestion 服务 (Node.js, ingestion-v2-combined)
    ↓ 处理事件、解析用户属性、写入
    ├── ClickHouse (事件数据、分析查询)
    └── PostgreSQL (用户画像、元数据)

PostHog Web 控制台
    ↓ 查询
    ├── ClickHouse (事件分析、漏斗、留存)
    └── PostgreSQL (项目配置、Feature Flags、仪表盘)
```

### 前端上报特性

- 异步上报，不阻塞页面渲染
- 事件先缓存在浏览器内存，批量发送
- 使用 `navigator.sendBeacon` 或异步 `fetch`
- 发送失败静默丢弃，不影响用户体验
- PostHog 服务短暂不可用时，仅丢失该时段数据，不影响业务

## 六、风险与应对

| 风险                               | 应对措施                                                 |
| ---------------------------------- | -------------------------------------------------------- |
| 服务器资源不足导致性能问题         | 初期申请 16GB 内存 + 50GB 磁盘，后续按数据量增长扩容     |
| PostHog 版本升级可能引入不兼容变更 | 锁定具体版本号，升级前在测试环境验证，做好数据库备份     |
| 单机部署存在单点故障               | 初期可接受，后续数据量大时可迁移到 Kubernetes 多副本部署 |
| 团队对 PostHog 不熟悉              | PostHog 文档完善，社区活跃，学习成本不高                 |

## 七、结论

PostHog 自托管方案在数据安全、功能完整性、性能、成本四个关键维度上都是最优选择，推荐采用。
